Relay agents

A relay is a lightweight agent you run inside your own network. It polls Lanby for assigned monitors and executes probes locally — letting you monitor services that are never exposed to the internet, without opening any inbound firewall ports or increasing your attack surface.

Why relays exist

Lanby's platform-managed probes run from the internet. That works great for public services, but breaks down the moment you want to check anything on a private network — a NAS, a home automation server, an internal database, or a service behind a VPN.

The naive solution — punch a hole in your firewall and let Lanby reach in — trades monitoring convenience for a permanent inbound exposure. A relay takes the opposite approach: the agent reaches out to Lanby. Your firewall stays exactly as it is.

Without a relay, Lanby can't reach services inside a private network. With a relay, the agent polls Lanby over outbound HTTPS and probes locally.

How it works

The relay agent runs as a Docker container on any machine inside your network. It maintains a persistent outbound connection to Lanby and polls for its assigned monitors. When a probe is due, the relay executes it locally and reports the result back.

The relay polls Lanby for due monitors, runs probes locally inside your network, then POSTs results back over HTTPS. A periodic heartbeat lets Lanby report relay health.

Security model

The relay is designed so that deploying it does not increase your network's attack surface. Every aspect of the architecture follows from that principle.

  • No inbound ports, ever. The relay does not listen on any port and does not run a web server. There is nothing reachable from outside your network — not from Lanby, not from anyone else. Your firewall rules do not change.
  • Outbound HTTPS only. All communication is initiated by the relay over port 443 using TLS. It's the same traffic profile as a browser or a package manager update — a single outbound HTTPS connection.
  • Probe results only, never payload data. The relay sends outcomes back to Lanby: pass/fail, latency, status code, and any error message. Response bodies and internal data stay inside your network. Lanby never sees your services' actual traffic.
  • No special privileges required. The relay container runs as an unprivileged process. No --privileged flag, no host networking, no kernel capabilities beyond what a normal container has.
  • Claim codes are one-time and account-scoped. A relay can only be claimed by someone who is logged in to your Lanby account and holds the code. The code is single-use and only printed to local logs — it cannot be discovered remotely.
ℹ️ The relay only needs outbound HTTPS (port 443) to reach Lanby, plus access to whichever internal hosts and ports you want it to probe. It does not need internet access for anything else.

The claim flow

When a relay starts for the first time it generates a unique identity and prints a short claim code to its logs. You paste that code into the Lanby console to associate the relay with your account. After claiming, the relay is ready to be assigned to monitors.

The claim code is shown once on first startup. After claiming, the relay is tied to your account and begins executing assigned monitors.

💡 The claim code is only printed on first startup. If you miss it, delete the identity volume (lanby-relay-data) and restart the container to generate a new one. The relay will need to be re-claimed.

Deploying a relay

1

Start the container

Run the relay with Docker. The identity volume keeps the relay's credentials across restarts.

docker run -d \
  --name lanby-relay \
  --restart unless-stopped \
  -v lanby-relay-data:/data \
  -e IDENTITY_PATH=/data/identity.json \
  ghcr.io/lanby-dev/lanby-relay:latest

Or with Docker Compose:

services:
  lanby-relay:
    image: ghcr.io/lanby-dev/lanby-relay:latest
    restart: unless-stopped
    environment:
      IDENTITY_PATH: /data/identity.json
    volumes:
      - relay-data:/data

volumes:
  relay-data:
2

Get the claim code from the logs

Once the container is running, the claim code is printed to stdout:

docker logs lanby-relay 2>&1 | grep "claim code"
# claim code: ABCD-1234

With Docker Compose:

docker compose logs lanby-relay 2>&1 | grep "claim code"
3

Claim the relay in the console

Go to console.lanby.dev/relays, paste the claim code into the Claim code field, and click Claim relay. The relay will appear in the table within a few seconds.

4

Assign monitors to the relay

When creating or editing a monitor, select your relay from the Relay dropdown. The relay will begin probing that target on its next poll cycle.

Identity and restarts

The relay generates a persistent identity file on first startup and saves it to the mounted volume. On every subsequent start, it loads the same identity — so the relay reconnects to your account automatically without needing to be re-claimed.

If you delete the identity volume or move the relay to a new machine without copying the identity file, the relay will generate a new identity and you will need to claim it again. The old relay entry in the console can then be deleted.

Requirements

The relay has minimal requirements:

  • Docker (any recent version)
  • Outbound HTTPS access to Lanby (port 443)
  • Network access to the services you want to probe
  • A persistent volume for the identity file

The relay has no inbound port requirements and does not run any web server.
